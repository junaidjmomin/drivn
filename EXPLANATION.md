# DRIVN: System Architecture & Engineering Analysis
*Authored by: Senior Full-Stack Engineer & MIT Professor of CS*

## 1. The Purpose: Urban Wellness Intelligence
The goal is to solve a specific problem: **Urban Stress.** We live in environments with high noise, poor air quality (AQI), and high-pressure commutes. DRIVN attempts to quantify this by correlating **biometric data** (Heart Rate, HRV, SpO2) with **environmental data** (AQI) to predict a user's stress level and provide actionable recommendations.

---

## 2. The Backend: The "Reasoning Engine" (`server.ts`)
In your original GitHub repo, you used a Python/Flask setup with a Random Forest model. In this implementation, I’ve shifted to a **Hybrid AI approach** using Node.js and Google’s Gemini 3 Flash.

*   **The Orchestrator**: We use **Express.js**. It’s the "waiter" of our application. It listens for requests on port 3000.
*   **The Brain (Gemini API)**: Instead of a static `.pkl` model, we use a Large Language Model (LLM). 
    *   *Why?* A Random Forest is great for classification, but an LLM can provide **contextual reasoning**. It doesn't just say "Stress is High"; it explains *why* (e.g., "High AQI combined with low HRV suggests environmental distress").
*   **The Protocol**: We use `responseSchema` in the Gemini config. This is critical. It forces the AI to return **deterministic JSON**, ensuring the frontend doesn't crash due to unexpected text formatting.

---

## 3. The Frontend: The "Cyberpunk HUD" (`src/App.tsx`)
The frontend is built with **React 19** and **Tailwind CSS 4**.

*   **State Management**: We use `useState` to track the `prediction` object. When the backend returns data, React detects the change in state and re-renders the UI.
*   **The Design System**: I implemented a "Cyberpunk" aesthetic using CSS variables and Tailwind.
    *   *Grid & Scanlines*: These are purely visual but serve a UX purpose—they signal a "high-tech, real-time" environment.
    *   **Motion (Framer Motion)**: We use `AnimatePresence`. When a new prediction arrives, the old UI doesn't just "snap" away; it fades and slides, providing a "crafted" feel.

---

## 4. How they are connected (The "Bridge")
In development, we use **Vite Middleware** inside Express. 
1.  When you run `npm run dev`, it starts `server.ts`.
2.  `server.ts` checks if it's in "development mode."
3.  If yes, it "invites" Vite to handle the frontend. 
4.  **The Result**: You have one server (Port 3000) handling both your API (`/api/predict`) and your React files. This eliminates "CORS" errors (Cross-Origin Resource Sharing) because the frontend and backend share the same origin.

---

## 5. Step-by-Step Execution Flow
1.  **Trigger**: User clicks "Run Prediction."
2.  **Simulation**: `generateRandomData()` creates a JSON object mimicking hardware sensors.
3.  **Transmission**: `fetch("/api/predict", ...)` sends a POST request to the Express server.
4.  **Processing**: The server receives the JSON, wraps it in a "System Instruction" prompt, and sends it to Gemini.
5.  **Inference**: Gemini calculates the stress score and generates recommendations.
6.  **Response**: Express sends the structured JSON back to the browser.
7.  **Visualization**: React updates the `prediction` state, triggering the animations you see on screen.

---

## 6. Critical Thinking: Issues & Limitations
1.  **Latency**: Sending biometric data to a cloud LLM takes ~1-2 seconds. For a real-time "emergency" stress monitor, this is too slow. A local ML model (like your original Python one) is faster.
2.  **Privacy**: Sending heart rate data to a third-party API (Google) is a massive privacy risk in a production medical app.
3.  **Cost**: Every click costs "tokens." A local model is free once trained.
4.  **Hallucination**: While the schema is strict, the "recommendations" are generated by an AI. They should never be taken as medical advice.

## 7. Deployment to GitHub
1.  **GitHub Pages won't work**: GitHub Pages only hosts static HTML/CSS. Since you have an Express backend, you need a **Platform as a Service (PaaS)** like **Render**, **Railway**, or **Fly.io**.
2.  **Environment Variables**: You **must** set your `GEMINI_API_KEY` in the dashboard of whatever hosting service you choose. Never commit your `.env` file to GitHub.
3.  **The Build Script**: Your `package.json` is already set up with `"build": "vite build"`. On a server, you would run `npm install`, then `npm run build`, then `npm start`.
